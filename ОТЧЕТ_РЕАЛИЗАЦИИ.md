# Отчёт о реализации: Оптимизация токенов и двухуровневый анализ

## 🎯 Цель
Кардинально снизить потребление токенов при тестировании путём внедрения двухуровневой системы анализа с кэшированием, батчингом и отслеживанием стоимости в реальном времени.

## ✅ Реализованные функции

### 1. Двухуровневая система анализа
- ✅ **Уровень 1 (Быстрый анализ)**: Использует `gpt-4o-mini` для быстрого, дешёвого первичного анализа договора
  - Стоимость: $0.15/$0.60 за 1М токенов (ввод/вывод)
  - Идеально для пакетной обработки всех пунктов
  - **В 66 раз дешевле** gpt-4-turbo

- ✅ **Уровень 2 (Глубокий анализ)**: Использует `gpt-4o` для детальной экспертизы по запросу
  - Стоимость: $2.50/$10.00 за 1М токенов
  - Включает:
    - Судебные прецеденты с номерами дел, судами, датами
    - Конкретные ссылки на законы (статьи ГК РФ)
    - Альтернативные формулировки с правовым обоснованием
    - Рекомендации экспертов и позиция ВС РФ
  - Пользователь выбирает 1-3 критичных пункта для глубокой проверки

**Сравнение стоимости:**
```
До оптимизации (gpt-4-turbo для всех):  ~$0.30-0.50 за договор
После (gpt-4o-mini + выборочно gpt-4o): ~$0.01-0.05 за договор
Экономия: снижение в 10-50 раз!
```

### 2. Пакетная обработка (`config/settings.py:llm_batch_size`)
Анализирует 5 пунктов за один запрос к LLM вместо 1 пункта за запрос.

**Преимущества:**
- System prompt отправляется 1 раз на 5 пунктов вместо 5 раз
- Сокращает общее количество API-вызовов в 5 раз
- **Дополнительная экономия 80%** на токенах system prompt

**Пример:**
```
20 пунктов до оптимизации: 20 API-вызовов
20 пунктов после: 4 API-вызова (5+5+5+5)
```

### 3. Кэширование ответов LLM
Кэширование на основе SHA256 в базе данных SQLite предотвращает повторные API-вызовы.

**Возможности:**
- Ключ кэша: хэш (провайдер, модель, промпт, system_prompt, температура, max_tokens, формат)
- Отслеживание количества попаданий и времени последнего доступа
- Автоматическое логирование попаданий в кэш
- **Стоимость:** $0 для закэшированных ответов

**Реализация:**
- Таблица `LLMCache` в `src/models/database.py`
- Методы кэширования в `src/services/llm_gateway.py`
- Включается параметром `use_cache=True`

### 4. Отслеживание токенов и отображение стоимости в реальном времени
Отслеживание каждого использованного токена и отображение стоимости в USD в интерфейсе.

**Отслеживаемые метрики:**
- Входные токены (промпт + system prompt)
- Выходные токены (ответ LLM)
- Всего токенов
- Стоимость в USD (рассчитывается по прайсу модели)
- Название модели

**Отображение в UI (`app.py:395`):**
```

📥 Input токенов: 1,234
📤 Output токенов: 567
💵 Стоимость: $0.0123
🤖 Модель: gpt-4o-mini
[Прогресс-бар, показывающий % от $0.10]
```

### 5. Настройки тестового режима
Переключение между тестовым и продакшн режимами для контроля затрат при разработке.

**Тестовый режим (`config/settings.py:llm_test_mode`):**
- Макс. токенов: 800 (против безлимита в продакшн)
- Макс. пунктов: 10 (против всех пунктов в продакшн)
- Модель: gpt-4o-mini по умолчанию

**Конфигурация:**
```python
llm_test_mode: bool = True
llm_quick_model: str = "gpt-4o-mini"
llm_deep_model: str = "gpt-4o"
llm_batch_size: int = 5
llm_test_max_tokens: int = 800
llm_test_max_clauses: int = 10
```

### 6. Интеграция глубокого анализа в UI
Добавлен раздел "🔬 Глубокий анализ (Уровень 2 - gpt-4o)" в веб-интерфейс.

**Возможности:**
- Виджет множественного выбора для выбора 1-3 критичных пунктов
- Кнопка "🚀 Запустить глубокий анализ"
- Детальное отображение результатов с:
  - Юридический анализ (оценка соответствия, исполнимости)
  - Риски с прецедентами (номера дел, суды, даты)
  - Альтернативные формулировки (2-3 улучшенных версии)
  - Рекомендации экспертов

**Расположение:** `app.py:574-661`

### 7. Улучшенная обработка JSON
Исправлена проблема, когда LLM возвращает текст вместо JSON.

**Исправления:**
- Более строгий system prompt, требующий JSON даже для тестовых данных
- Попытка извлечь JSON из текстового ответа через regex
- Fallback на индивидуальный анализ при ошибке батча
- Импорт `settings` в `contract_analyzer_agent.py`

## 🚀 Как использовать

### 1. Включить тестовый режим (для разработки)
В `config/settings.py`:
```python
llm_test_mode = True  # Ограничивает токены и пункты
```

```bash
./start_all.sh
```

- Нажмите "🔍 Начать анализ"
- Увидите использование токенов и стоимость в UI

- После базового анализа прокрутите до "🔬 Глубокий анализ"

- Нажмите "🚀 Запустить глубокий анализ"
- Изучите детальный экспертный анализ с прецедентами

### 4. Проверить эффективность кэша
```bash
python3 create_cache_table.py
```

## 🔧 Исправленные проблемы

### Проблема 1: LLM возвращал текст вместо JSON
**Ошибка:**
```
Failed to parse JSON response: Expecting value: line 1 column 1 (char 0)
Raw response: Для проведения анализа необходимо предоставить реальный текст...
```

**Причина:** LLM отказывался анализировать тестовый текст и просил реальные данные

**Решение:**
1. Улучшен system prompt с требованием ВСЕГДА возвращать JSON
2. Добавлена попытка извлечь JSON из текста через regex
3. Реализован fallback на индивидуальный анализ при ошибке батча

### Проблема 2: settings не импортирован
**Ошибка:**
```
name 'settings' is not defined
```

**Решение:** Добавлен импорт `from config.settings import settings` в начало файла `contract_analyzer_agent.py`
